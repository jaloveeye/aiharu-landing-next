# Skill: preflight-review (작업 전 점검 & 개선 제안)

## 목표
- 어떤 작업이든 **착수 전에** “버그/확장성/컨벤션/보안/모호함”을 빠르게 점검하고,
  사용자에게 **요약 보고 + 개선안**을 제시한 뒤 진행합니다.

## 트리거
- 사용자가 “명령/요청/지시” 형태로 작업을 시키는 경우(코드 수정, 설계, 운영 작업 포함)
- 특히, 영향 범위가 크거나(배포/DB/권한/결제), 변경이 넓거나(리팩터링), 안전이 중요한 작업

## 수행 절차 (항상 이 순서)
1) **Preflight Review(보고)**  
   아래 4항목을 3~8줄로 요약해 보고합니다.
   - 버그 가능/확장 문제: 깨질 구간, 엣지케이스, 유지보수 부담
   - 컨벤션/일반성: 흔치 않은 패턴, 팀 표준 위배 가능성, 읽기/테스트 난이도
     - (코드 변경 시) 브랜치 전략/PR 정책이 Git Flow 규칙을 위배하지 않는지
   - 보안/리스크: 비밀값, 권한, 입력 검증, 민감정보 로그, 데이터 파괴 가능성
   - 모호함/누락: 가정이 필요한 부분, 성공 기준, 범위(Scope) 불명확

2) **개선안 제시(선택지 1~3개)**  
   - “안전한 기본안” + “빠른 안” + “확장 가능한 안”처럼 옵션으로 제시하면 좋습니다.

3) **진행 방식**
   - 사용자 답변이 꼭 필요하지 않으면: **가장 안전한 기본안으로 바로 진행**
   - 답변이 없으면 실패할 수 있는 필수 정보가 있으면: **필수 질문 1~2개만** 묻고 진행
   - 시간/토큰이 빠듯하면: 보고를 **초압축(4줄)**으로 하고 진행
